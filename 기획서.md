# 워프레임 마우스 민감도 추천 시스템 기획서

## 1. 프로젝트 개요
- **이름(가칭):** WarSens (Warframe Sensitivity Advisor)
- **목표:** 사용자의 실제 플레이 데이터와 신체적 사용 범위를 기반으로, **개인 맞춤 최적 마우스 민감도(절대값)** 를 추천하는 AI 기반 데스크톱 프로그램
- **개발 형태:** 1인 개발, Windows 데스크톱, 기간 무제한

## 2. 문제 정의
- 워프레임은 파쿠르, 에임글라이드, 360도 빠른 시점 전환이 잦아 일반 FPS 민감도 계산기(cm/360 기반 계산기 등)로는 최적값 판단이 어렵다.
- 사용자마다 **마우스 사용 면적(손목/팔꿈치/어깨 움직임)**, DPI, 마우스패드 크기가 달라 일률적 값 추천이 무의미하다.
- 기존 툴들은 "입력 → 공식 계산"에 그치고, **실제 플레이 결과로 피드백 루프를 돌리는 추천 시스템은 부재**하다.

### 2-1. 유사 선례 및 차별점
조사한 유사 시도(2026-04 기준):

| 사례 | 접근 | 한계 |
|---|---|---|
| **Chimpshot** (상용) | 마우스 로그 → 민감도 추천 | 범용 FPS, 영상 분석 없음, 폐쇄형 |
| **Aimlabs Mouselabs** | 에임 트레이너 성적 → 민감도 제안 | 훈련 환경 한정, 실게임 궤적 미반영 |
| **Oblivity** (Steam) | 3D 맵 에임 테스트 → 민감도 | 훈련 게임, 추천은 휴리스틱 |
| **Ngambarde / aim_trainer_analysis** (OSS) | Kovaak's 녹화에 YOLOv8 적용 | 분석까지만, 추천 파이프라인 없음, 단일 게임 한정 |
| **NVIDIA 2022 "Esports Sensitivity" 논문** | 실험실 환경 최적 민감도 분석 | 연구 단계, 실사용 제품 아님 |

**WarSens의 차별점 3가지:**
1. **워프레임 전용 도메인 지식** — 파쿠르/에임글라이드/360 전환이 섞인 궤적을 그대로 분석
2. **영상 + 입력 로그 결합** — CV로 추출한 적 좌표와 마우스 델타를 프레임 단위 동기화
3. **End-to-end 파이프라인** — 수집 → 분석 → 절대값 추천까지 한 앱에서 완결

### 2-2. 학습된 함정 (사전 조사 결과)
- **개인 분산이 크다** — 동일 사용자도 일관된 최적값이 아닌 범위를 가진다. 출력은 단일값 + **신뢰구간**으로 제시.
- **사용자는 실제로 민감도를 잘 바꾸지 않는다** — 추천만 던지면 채택률이 낮다. UI에 **A/B 비교 루프**(현재값 vs 추천값 짧은 세션)를 내장.
- **YOLO는 게임 간 일반화가 어렵다** — 단일 타이틀(워프레임) 집중이 오히려 옳은 선택.

## 3. 솔루션 개요
사용자의 실제 게임플레이에서 **조준 궤적(aim trajectory)** 을 추출·분석하여, 조준 효율성이 최대화되는 민감도를 회귀적으로 추정한다.

- **입력:** 현재 민감도, DPI, 마우스 입력 로그, 화면 녹화(→ YOLO로 적 위치 추출)
- **중간 지표:** 타겟 도달 시간, 오버슈트 비율, 마우스 이동 분포(사용 면적 추정)
- **출력:** 최적 민감도 절대값 (예: "인게임 민감도 0.42 권장")

## 4. 시스템 아키텍처

```
[클라이언트 (Windows 데스크톱 앱)]
  ├─ 마우스 입력 로거 (raw input hook, 타임스탬프+dx/dy)
  ├─ 화면 녹화 모듈 (게임 창만 캡처)
  ├─ YOLO 추론기 (로컬, 적·크로스헤어 위치 추출)
  ├─ 전처리기 (영상 → "프레임별 적 좌표 + 조준점 좌표 + 마우스 델타" 시계열)
  └─ 업로더 (가공된 JSON/Parquet만 서버로)

[서버 (추후 필요 시)]
  ├─ 수집 데이터 저장소
  ├─ 추천 모델 학습/배포
  └─ 추천 API

[사용자 앱 UI]
  ├─ 캘리브레이션/측정 세션 런처
  ├─ 분석 대시보드 (오버슈트율, 평균 도달시간 등)
  └─ 추천 민감도 표시 + 근거 설명
```

핵심 원칙: **영상 원본은 절대 서버로 보내지 않는다.** 로컬에서 좌표 시계열로 변환한 뒤 가공 데이터만 업로드.

## 5. 주요 모듈

### 5-1. 데이터 수집 클라이언트 (가장 큰 공수)
- **마우스 로거:** Windows Raw Input API 사용, 1000Hz 정밀 기록
- **화면 녹화:** OBS 대신 내장(DXGI Desktop Duplication) — 워프레임 창만 타겟
- **동기화:** 마우스 이벤트와 프레임을 타임스탬프로 정렬
- **YOLO 추론:** 사용자 PC에서 실시간 또는 후처리 (후처리가 부담 적음)

### 5-2. YOLO 커스텀 모델
- **클래스:** Grineer, Corpus, Infested, Sentient (대분류 먼저), 크로스헤어
- **데이터:** 본인이 시뮬라크럼에서 다양한 적 스폰 후 스크린샷 수집 + 라벨링
- **대안:** 크로스헤어는 YOLO 대신 템플릿 매칭으로 탐지 (고정 위치 + 모양 일정)

### 5-3. 조준 궤적 분석
프레임별 `{적 위치, 크로스헤어 위치, 마우스 델타}` 시계열에서:
- **타겟 도달 시간:** 새 적 등장 → 크로스헤어가 적 히트박스 진입까지 ms
- **오버슈트율:** 타겟 지나쳐서 되돌아온 횟수 / 총 조준 시도
- **마우스 사용 면적:** 세션 내 누적 dx/dy의 2D 분포 (표준편차, 95퍼센타일 반경)

### 5-4. 추천 모델
- **베이스라인:** 회귀 (XGBoost 또는 MLP) — 입력: 사용자 특성 + 현재 민감도 + 궤적 지표 / 출력: "최적 민감도 - 현재 민감도" 델타
- **데이터가 쌓이면:** 같은 사용자의 여러 민감도 세션으로 **개인화 미세조정**
- **왜 DL 고집 안 함:** 데이터가 수백~수천 세션 수준이면 tabular ML이 더 안정적. 궤적 자체를 인코딩해야 하는 단계에서 DL(1D CNN, Transformer)로 확장 가능.

## 6. 데이터 수집 전략

### 6-1. 벤치마크 세션 (표준화, 필수)
- 시뮬라크럼에서 **고정 루틴** 수행 (예: Grineer Lancer 10마리 스폰 → 헤드샷 시도 5분)
- 참여자가 **동일 조건**에서 측정 → 비교 가능 데이터

### 6-2. 일반 플레이 세션 (보조)
- 백그라운드 로깅으로 실전 데이터 수집
- 미션 종류/적 종류를 메타데이터로 태깅

### 6-3. 참여자 모집 및 보상
- **1차 모집단:** 본인 + 오픈채팅방
- **보상:** 플래티넘 직접 지급 + 비금전적 보상 (초기 사용권, 결과 리포트 공유, 크레딧 표기)
- **리스크:** 플래티넘 거래는 세금 및 거래 쿨다운 존재 → 보상 규모와 예산을 사전에 산정 필요

## 7. 기술 스택 (안)

| 영역 | 선택 |
|---|---|
| 언어 | Python 3.11+ |
| UI | PyQt6 또는 PySide6 |
| 마우스 후킹 | `pynput` + Win32 Raw Input (정밀도 필요 시) |
| 화면 캡처 | `dxcam` (DXGI 기반, 고속) |
| CV 모델 | YOLOv8 또는 YOLOv11 (Ultralytics) |
| ML | PyTorch + scikit-learn/XGBoost |
| 데이터 포맷 | Parquet (시계열), JSON (메타데이터) |
| 서버 (추후) | FastAPI + SQLite/PostgreSQL |

## 8. 개발 로드맵

**Phase 0 — 프로토타입 (혼자 검증)**
1. 마우스 로거 + 화면 캡처 동기화 성공
2. YOLO로 시뮬라크럼 영상에서 적·크로스헤어 추출 성공
3. 본인 플레이 2~3가지 민감도로 녹화 → 오버슈트율 차이 확인

**Phase 1 — MVP**
4. UI로 세션 시작/종료 + 간이 대시보드
5. 단일 사용자(본인) 기준 추천 로직 (단순 휴리스틱: 오버슈트율 ↑ → 민감도 ↓)
6. 설치 가능한 exe 빌드

**Phase 2 — 데이터 수집 공개**
7. 오픈채팅방 베타 테스트 (5~10명)
8. 가공 데이터 업로드 파이프라인
9. 수집 데이터로 ML 모델 학습

**Phase 3 — 추천 정식 서비스**
10. 개인화 추천 + 근거 설명 UI
11. 신규 사용자 온보딩 (캘리브레이션 가이드)

## 9. 리스크 및 대응

### 9-1. 안티치트(EAC) — 기술 조사 결론
- EAC는 **게임 프로세스 메모리 접근 / DLL 인젝션 / 입력 주입**을 감지한다.
- **OS 레벨 Raw Input 폴링**과 **DXGI Desktop Duplication 화면 캡처**는 OBS/디스코드 오버레이와 동일 카테고리로, 감지 대상이 아니다.
- 따라서 기술적 밴 리스크는 낮음. 단, EULA 상 "unfair advantage" 해석은 회색지대이므로 **자동 보조(aimbot/트리거봇) 금지선**을 정책으로 못박는다.

**MUST (반드시 지킨다):**
- 마우스 입력은 **읽기 전용** — Windows Raw Input API 또는 `pynput.mouse.Listener`(수신 전용)
- 화면 캡처는 **DXGI Desktop Duplication** (`dxcam`) 또는 동급 — 게임 프로세스에 접근 금지
- CV/분석 결과는 **오프라인 리포트·추천**으로만 사용
- 사용자에게 동작 내용을 **명시적으로 고지**(앱 내 동의 화면)

**MUST NOT (절대 금지):**
- 입력 주입: `pynput.mouse.Controller`, `pyautogui`, `SendInput`, `mouse_event`, `Interception` 드라이버
- 게임 프로세스 접근: `OpenProcess`, `ReadProcessMemory`, `WriteProcessMemory`, Cheat Engine 류
- DLL 인젝션, 메모리 훅, 그래픽스 API 훅(DirectX/Vulkan)
- CV 탐지 결과가 **실시간으로 입력에 피드백되는 루프** (사실상 aimbot)

상세 정책: [docs/SAFETY_POLICY.md](docs/SAFETY_POLICY.md).

### 9-2. 리스크 테이블

| 리스크 | 대응 |
|---|---|
| **데이터 수집 툴이 전체 공수의 60~70%** | Phase 0에서 먼저 검증, 툴 자체를 별도 부산물로 공개 가치 있음 |
| 워프레임 EULA/안티치트(EAC) 이슈 | §9-1 MUST/MUST NOT 준수. 의심 시 공식 포럼/DE 문의 |
| YOLO 라벨링 노동 | 초기엔 크로스헤어(템플릿 매칭)만으로도 오버슈트 측정 가능 → 적 탐지는 후순위 |
| 참여자 데이터 편향 (고수 위주, 저사양 PC 제외 등) | 최소 표본 수 확보 전까지는 "개인화 추천"만 제공, 범용 추천은 홀드 |
| 플래티넘 보상 비용 폭증 | 참여 할당량 설정 (예: 1인당 벤치마크 3세션까지만 보상) |
| "좋은 민감도"의 정답이 없음 — 사용자 취향 문제 | 객관 지표(궤적 효율) + 주관 평가(간단 설문) 병행. 출력은 **단일값 + 신뢰구간**, UI에 **A/B 비교**로 채택률 보정 |
| 선행 사례 존재 (§2-1) | 워프레임 전용 + 영상+입력 결합 + end-to-end 파이프라인의 조합으로 차별화 |
