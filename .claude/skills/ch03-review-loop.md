# ch03 — 리뷰 루프 프로토콜

> 작성자 ↔ 감사관 ↔ 오케스트레이터 3자 간의 합의 규칙. orchestrator 에이전트가 이 프로토콜을 실행한다.

## 적용 대상
- 코드 작성/수정 (→ code-auditor)
- 기획서 수정 / 스펙 변경 (→ plan-auditor)
- 복잡한 멀티 도메인 작업 (→ 두 감사관 모두)

## 루프 프로토콜

### Iteration 진행 순서
1. 오케스트레이터 → 작성자: 원본 지시 + (2회차부터) 감사관 제안
2. 작성자 → 오케스트레이터: 작업 보고서 + 변경 파일 목록
3. 오케스트레이터 → 감사관: 작성자 보고서 + 변경 파일 경로
4. 감사관 → 오케스트레이터: 감사 보고서 (PASS / CONDITIONAL PASS / FAIL + 제안 목록)
5. 오케스트레이터: 판정
   - PASS → 종료
   - 그 외 → 다음 Iteration

### 최대 반복 횟수: **5회**
- Iteration 1~4: 자동 루프
- Iteration 5 완료 후에도 PASS 아니면 **사용자 에스컬레이션**

## 판정 상세 기준

### PASS
- 감사관이 `PASS` 반환
- 또는 `CONDITIONAL PASS` + 심각도 `낮음`만 + 작성자가 사소한 개선으로 판단

### CONDITIONAL PASS (루프 계속)
- 심각도 `중간` 이하 문제
- 작성자가 수용 가능한 제안

### FAIL (루프 계속)
- 심각도 `높음` 문제 존재 (보안, 데이터 손실 가능성, 기획 위반)
- **반드시 수정 후 재감사**

## 작성자 반박 규칙
작성자가 감사관 제안을 거부할 수 있는 조건:
- 기획서와 정면 배치되는 제안
- 감사관이 컨텍스트를 오해한 경우
- 성능/가독성 측면에서 명확한 근거가 있는 경우

반박 시:
1. 작성자는 **반박 근거를 보고서에 명시**
2. 오케스트레이터가 반박을 다음 감사 요청에 **원문 그대로 첨부**
3. 감사관이 반박을 수용하면 해당 제안은 다음 판정에서 제외
4. 감사관이 반박을 기각하면 쟁점으로 기록

## 에스컬레이션 트리거
1. Iteration 5 완료 후 PASS 실패
2. 동일 쟁점이 Iteration 3회 이상 반복
3. 작성자-감사관이 기획서 해석에서 충돌 (오케스트레이터가 결정 불가)

## 사용자에게 중간 알림
오케스트레이터는 각 Iteration 시작 시 간결히 알림:
```
[Iteration 2/5] code-auditor 지적사항 반영 중 — data-collector-agent 수정 작업
```

루프 진행 중 상세 내용은 드러내지 않음. 최종 보고 또는 에스컬레이션에서만 전체 공개.

## 적용 예시

### 예: 마우스 로거 PoC 작성
- 오케스트레이터 → `data-collector-agent`: "Raw Input 기반 마우스 로거 작성"
- `data-collector-agent` → 보고서 (src/collector/mouse_logger.py 작성됨)
- 오케스트레이터 → `code-auditor`: "이 파일 검증"
- `code-auditor` → 보고서 (심각도 중간 2건: 에러 처리 누락, 리소스 정리 누락)
- Iteration 2: `data-collector-agent` 수정 → `code-auditor` 재감사 → PASS
- 오케스트레이터 → 사용자: 최종 보고 (Iteration 2회, 수정 내용 요약)
